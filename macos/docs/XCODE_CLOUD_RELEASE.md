# Xcode Cloud ‚Äî Release Pipeline Runbook (macOS ‚Üí TestFlight)

How "merge to `master` ‚Üí TestFlight, gated on a passing test run" is configured.
This is the **build/sign/distribute** layer; the GitHub Actions `ci.yml` PR gate and
the nightly Developer-ID DMG are separate (see *How it composes* at the end).

> Most of this is **clicked in App Store Connect** (Xcode Cloud workflows live there,
> not in the repo). The repo already carries the `ci_scripts/` hooks Xcode Cloud needs.
> The one repo-side change worth making ‚Äî the **post-archive MLX smoke** ‚Äî is in ¬ß4.

---

## 1. Where Xcode Cloud sits

| Layer | Tool | Trigger | Job |
|---|---|---|---|
| PR gate | GitHub Actions `ci.yml` | every PR | `swift test` (macos-26) + Python smoke + Claude review ‚Äî fast |
| **Release** | **Xcode Cloud** | push to `master` | Test ‚Üí Archive ‚Üí **TestFlight** |
| Nightly DMG | GitHub Actions cron | nightly | Developer-ID notarized `.dmg` ‚Üí GitHub Release |

Xcode Cloud manages signing natively (cloud signing via your ASC API key) ‚Äî **no
certificates or notary credentials live in this pipeline.** Those only matter for the
separate Developer-ID DMG path (`tools/release/release-macos.sh`).

## 3. The "Release" workflow (App Store Connect ‚Üí Xcode Cloud)

**Xcode (or App Store Connect) ‚Üí your app ‚Üí Xcode Cloud ‚Üí Manage Workflows ‚Üí New Workflow.**

### General
- **Name:** `Release`
- **Repository:** this repo ¬∑ **Primary branch:** `master`
- **Xcode:** *Latest Release* (must resolve to **Xcode 26+** ‚Äî the suite links
  `FoundationModels.framework`, a macOS 26 runtime framework; an older toolchain fails).
- **macOS:** latest available image.

### Start Conditions
- **Branch Changes** ‚Üí **`master`** (i.e. on every merge/push to master).
- *(Optional)* **Files and Folders ‚Üí Custom:** skip docs-only churn. Suggested
  "don't build" globs: `**/*.md`, `marketing/**`, `site/**`, `docs/**`,
  `**/project-memory.md`. (Keeps a README merge from minting a TestFlight build.)

### Actions ‚Äî ordered; a failed action stops the workflow
1. **Test**  ‚Üê *the gate*
   - **Scheme:** `M1K3`  ¬∑  **Platform:** macOS  ¬∑  **Destination:** Mac (Any)
   - This runs the same SwiftPM tests as the PR gate. If it fails, **Archive never runs**,
     so nothing reaches TestFlight. That is the "successful test gate."
2. **Archive**
   - **Scheme:** `M1K3`  ¬∑  **Platform:** macOS
   - **Distribution:** *TestFlight (Internal Testing) and App Store*

### Post-Actions
- **TestFlight Internal Testing** ‚Üí your internal tester group.
  Internal testing needs **no Beta App Review**, so builds land in minutes and the blast
  radius is just your team. (Promote to an **External** group later, once the gate has
  earned trust ‚Äî external adds Beta App Review.)

### Build number
- Enable **"Automatically manage build numbers"** (Xcode Cloud increments per build), so
  every master merge produces a uniquely-numbered TestFlight build without a repo commit.
  `MARKETING_VERSION` stays the single source of truth in `macos/project.yml`.

## 4. Closing the gate's blind spot ‚Äî the post-archive MLX smoke ‚öÝÔ∏è

**The Test action above has a hole.** `swift test` runs with the MLX / Metal / WhisperKit
integration tests **gated OFF** (`M1K3_MLX_INTEGRATION=0` in `ci.yml`; the "metallib wall"
‚Äî those paths only run from a bundled `.app`, never the CLI test binary). So a regression
in the **on-device brain or voice path can pass the gate and still ship to TestFlight.**

The app already ships the fix: a **self-test harness** (`macos/M1K3App/SelfTest.swift`)
that runs *inside the built `.app`* and exercises the real GPU/MLX path:

```bash
# Streams each stage to stderr; writes a structured log to M1K3_SELFTEST_OUT.
M1K3_SELFTEST=1 M1K3_SELFTEST_TTFT=1 \
  /path/to/M1K3.app/Contents/MacOS/M1K3 2> /tmp/m1k3_selftest.log
```

Real stages (each opt-in by its own flag): `TTFT`, `CHATEVAL` (full brain eval ‚Äî tool-use,
reasoning), `MEMGRAPH`, `MEMEVAL`, `MEMLOOP`, `KVPERSIST`, `ABSEP`. Model id via
`M1K3_SELFTEST_MODEL`.

**Wire it as a post-build hook** so a failed self-test fails the Xcode Cloud build *before*
TestFlight distribution. Drop this into `ci_scripts/ci_post_xcodebuild.sh` (sketch ‚Äî see
"open decisions" before committing):

```bash
# Only after a successful ARCHIVE.
if [ "${CI_XCODEBUILD_ACTION:-}" = "archive" ] && [ "${CI_XCODEBUILD_EXIT_CODE:-1}" = "0" ]; then
  APP=$(find "${CI_ARCHIVE_PATH:-$CI_AD_HOC_SIGNED_APP_PATH}" -name 'M1K3.app' -maxdepth 4 | head -1)
  if [ -n "$APP" ]; then
    echo "--- MLX smoke: $APP"
    M1K3_SELFTEST=1 M1K3_SELFTEST_TTFT=1 M1K3_SELFTEST_OUT=/tmp/m1k3_selftest.log \
      "$APP/Contents/MacOS/M1K3" 2>&1 | tail -40 || true
    if grep -qiE 'FAIL|error' /tmp/m1k3_selftest.log; then
      echo "‚ùå MLX self-test FAILED ‚Äî blocking TestFlight"; exit 1
    fi
    echo "‚úÖ MLX self-test passed"
  fi
fi
```

**Open decisions before this is live (the model-in-CI problem):**
- **GPU:** Xcode Cloud runs Apple-Silicon VMs (Metal present) ‚Äî but confirm MLX initialises
  on the image; if not, the smoke must be skipped there and run on the nightly DMG runner instead.
- **Model availability:** the self-test needs a model. Either point `M1K3_SELFTEST_MODEL` at a
  **tiny eval model** (fast, cheap) or accept a one-time download per build (slower, costs minutes).
  Pick the tiny-model route to keep release builds fast.
- **Timeout:** budget the smoke against the Xcode Cloud action timeout; `TTFT` alone is the
  cheapest meaningful signal ‚Äî start there, add `CHATEVAL` only if the budget allows.

## 5. Verification & rollback

- **First run:** merge a trivial change to `master` ‚Üí watch the Xcode Cloud build: Test ‚úÖ ‚Üí
  Archive ‚úÖ ‚Üí appears in TestFlight ‚Üí installable by the internal group.
- **Prove the gate:** push a deliberately-failing test on a throwaway branch merged to a
  scratch base ‚Üí confirm Archive/TestFlight **do not** run.
- **Rollback:** TestFlight builds can be **expired** in App Store Connect; internal-only blast
  radius means a bad build never reaches the public. There is no "unship" ‚Äî expire + ship the fix.

## 6. How it composes

- **PR** ‚Üí GitHub Actions `ci.yml` (fast gate; merge-blocker).
- **Merge to `master`** ‚Üí Xcode Cloud `Release` (Test ‚Üí Archive ‚Üí TestFlight + MLX smoke).
- **Nightly** ‚Üí GitHub Actions cron ‚Üí Developer-ID notarized `.dmg` ‚Üí rolling `nightly` GitHub
  Release (the off-store / Homebrew-cask path; separate signing creds, not this pipeline).
- **Metadata** ‚Üí `fastlane mac metadata` (`deliver`) ‚Äî text/screenshots only, run when copy changes.
