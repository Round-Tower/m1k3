# 0002. Pin and verify downloaded model weights; refuse on mismatch rather than heal

Date: 2026-07-21
Status: ACCEPTED
Deciders: Kev + claude-opus-4-8

## Context

Every brain M1K3 runs is fetched at runtime from a third-party host. Until this
decision, that fetch was unpinned and unverified:

- `BrainWeightsFetcher.swift` passed `revision: nil`, which
  `HuggingFaceBridge.download` turned into `"main"`.
- Whatever `main` pointed at, at the moment a given user happened to onboard,
  was downloaded and handed to MLX.
- Nothing checked the bytes. MLX's safetensors loader `malloc`s and reads
  (`load.cpp`), so the weights go straight into unified memory and execute.

That is a live supply-chain exposure with three distinct triggers: the host
being compromised, an `mlx-community` org account being hijacked, or a repo
being force-pushed. It is also not hypothetical drift: the 2026-07-09 gemma
chat-template fix was expected to reach us *purely via a weights re-pull*, with
no code change and no review on our side. Upstream content already moves under
us; we simply had no way to see it.

The prompting concern was availability — a single third-party host is a single
point of failure during onboarding, which is the worst possible moment. But
availability and integrity are different threats with different fixes, and
conflating them is how you end up with a second CDN that makes things worse:
**two sources you don't verify is two ways in.** Integrity has to come first,
because verification is what makes a second source safe to add at all.

## Decision

Two mechanisms, both required:

**1. Pin the revision.** `HuggingFaceBridge.download` resolves an absent
revision to the pinned commit SHA for that repo, not to `main`. An explicit
caller-supplied revision still wins, so the eval harness and A/B overrides are
unaffected.

**2. Verify the bytes against a manifest committed to this repo.**
`PinnedWeights.swift` carries sha256 digests and sizes for every file we
download. It is generated by `tools/weights/pin_weights.py` from a local
snapshot that has actually been run and evaluated, then cross-checked against
HuggingFace's published LFS oid; the generator hard-stops on disagreement.

The manifest must live in our source. Fetching the expected digest from the
same host that serves the file proves nothing — whoever can swap the file can
swap its published hash. A manifest that travels with the code, under review,
is the only version of this that is not theatre.

### Enforcement point

Verification runs inside `HubApiDownloader.download`, the single choke point
every fetch passes through — including `loadContainer`'s own, because the MLX
factories take our downloader as their `Downloader`. That placement is
load-bearing: it is after the snapshot completes but before any factory reads a
tensor, which is the only window in which refusing actually prevents a poisoned
load. Checking *before* the download would pass trivially on a first run (empty
directory) and leave onboarding — the case this exists to protect — unguarded.

All three of `download`'s exits are covered, including the offline and
auth-required fallbacks that return a pre-existing local copy. A cache poisoned
on an earlier run must not become trusted merely because the network is down.

### Refuse, do not heal

`ModelCacheIntegrity` (ADR-less, 2026-07-16) heals a torn cache silently:
delete, re-download, carry on. `WeightIntegrity` deliberately does **not**.

A torn cache is an accident. A digest mismatch is evidence. Silently
re-downloading from the same poisoned source would spin an infinite
re-download loop that hides an active attack behind what looks like flaky
networking. So a mismatch throws `WeightTamperError`, leaves the files in place,
and logs `.fault`. The error is not a transient network error, so `withRetry`'s
default `isTransientNetworkError` will not match it and the load fails to
`AppReadiness.failed` instead of looping.

### Absent is not tampered

Mid-download, most pinned files are absent — overwhelmingly the common case.
Treating that as an attack would alarm on every first run. Absent is
`.incomplete`; present-but-wrong is `.tampered`. Tampering outranks
incompleteness, so a half-downloaded directory containing one wrong file still
alarms.

### Unpinned repos still load

Repos absent from the manifest return `.unpinned` and load normally. Pinning
protects what we *ship*; it is not an allowlist. Spike checkpoints and
`M1K3_SELFTEST_CHATEVAL_MLX_MODEL` A/B overrides must stay loadable or the
evaluation loop — which is how both brains got chosen — becomes unusable.

### Cost

Hashing ~6.7 GB on every launch would be indefensible, so a verified directory
writes a receipt naming the revision and the sizes it verified; a later load
with the same revision and unchanged sizes skips hashing. This is sound under
the stated threat model — a poisoned *download*, not local tampering. An
attacker who can already write inside our sandbox container owns the app
regardless of what we hash.

## Consequences

**Good.** Onboarding no longer trusts a moving target. An upstream change to a
shipped model becomes a visible, reviewed decision instead of a silent one. The
retrieval embedder is covered by the same choke point at no extra cost. And
because we now know what correct looks like, a second download source can be
added safely — which is the next piece of work.

**Cost.** Promoting a brain now requires regenerating the manifest. This is
deliberate friction on a step that changes what users execute, and the
generator exists to keep it mechanical: both tiers moved inside 26 hours in
July 2026, so a manual process would have been skipped.

**Bootstrapping.** Pinning is seeded from bytes we already trusted — the
snapshots that produced the 2026-07-15/16 eval results, independently confirmed
against HuggingFace's published LFS oids for all seven large files. That is the
irreducible trust-on-first-use step every pinning scheme has. It is narrowed,
not eliminated.

**Not yet covered.** Kokoro TTS weights are fetched via raw
`resolve/main/` URLs in `KokoroSpeechProvider.swift`, bypassing
`HubApiDownloader` entirely. That path needs its own work and is tracked
separately.
