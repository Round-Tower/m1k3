
# Kotlin Multiplatform AI Companion App Architecture: Comprehensive 2025 Implementation Guide

**Apple Intelligence integration is now production-ready with the Foundation Models framework requiring just 3 lines of Swift code**, while NNAPI deprecation in Android 15 fundamentally shifts the Android ML landscape toward TensorFlow Lite in Play Services. This report provides actionable architectural guidance for building privacy-first, on-device AI apps that maintain feature parity across iOS and Android while achieving 95%+ accessibility compliance.

For immediate implementation, **target iOS 15+ and Android API 27+ as minimum versions** (96% combined market coverage), use Core ML 5 with MLProgram format on iOS alongside TensorFlow Lite GPU delegates on Android, implement expect/actual patterns for platform-specific AI engines, and build graceful degradation chains with device tier classification (Flagship 8GB+, Mid-range 4-8GB, Budget <4GB). The Foundation Models framework delivers comparable performance to 7B parameter models despite being only 3B parameters through aggressive quantization and Neural Engine optimization, while Apple's Private Cloud Compute architecture provides cryptographically verifiable privacy for complex tasks requiring server-side processing.

## Apple's Foundation Models framework reshapes iOS AI development

The Foundation Models framework represents Apple's most significant AI developer API release, providing direct access to the **~3 billion parameter on-device language model** powering Apple Intelligence. Available in iOS 18.1+ for devices with A17 Pro/M1 chips or later, the framework delivers text understanding, summarization, entity extraction, and structured output generation with **zero API costs** and complete offline functionality.

**Basic implementation requires minimal code.** A simple text processing workflow initializes a LanguageModelSession and streams responses asynchronously. The framework supports guided generation for structured outputs using the @Generable macro with @Guide property annotations, enabling reliable extraction of typed data structures from natural language. For conversational AI, stateful sessions maintain transcript history automatically, while tool calling capabilities allow models to invoke app-specific functions. Temperature control via GenerationOptions provides deterministic output with greedy sampling or creative variety with random sampling.

**The architecture employs aggressive optimization techniques.** Apple's foundation model uses shared input/output embedding matrices for memory efficiency, grouped-query attention for reduced memory bandwidth, and RoPE embeddings for long-context support. Compression reaches **2-bit average precision** through quantization-aware training and mixed-precision techniques (3.5 bits average), with ASTC texture compression for weight storage and 4-bit quantized embedding tables. Runtime-swappable LoRA adapters enable task-specific specialization without maintaining separate models. Despite compression, benchmarks show the on-device model **outperforms Mistral-7B and Gemma-7B** (both larger at 7B parameters) in human preference evaluations, demonstrating that aggressive optimization combined with Apple Silicon's Neural Engine produces competitive results at significantly smaller model sizes.

**Private Cloud Compute handles requests exceeding on-device capacity** while maintaining Apple's privacy guarantees. The system analyzes each request on-device and triggers PCC when complexity requires larger server models. PCC nodes run custom Apple Silicon with Secure Enclaves and verified boot chains, processing requests in **stateless execution with cryptographic erasure** after response completion. Request metadata excludes personally identifiable information, OHTTP relays hide source IP addresses, and RSA blind signatures provide single-use credentials preventing request correlation. All production software images publish to an append-only transparency log within 90 days, enabling independent security research through the Virtual Research Environment.

**Integration with custom models requires parallel implementation patterns.** Foundation Models cannot directly combine with custom models like Gemma, requiring separate execution paths. Use Foundation Models for text understanding, summarization, and natural language generation while deploying custom models via Core ML for specialized tasks like domain-specific classification or computer vision. Convert Gemma or other Hugging Face models to Core ML format using coremltools with quantization options, then configure MLModelConfiguration to use all compute units (Neural Engine, GPU, CPU). For cross-platform requirements, implement ONNX Runtime on iOS with CoreML execution provider, though Core ML offers better optimization for Apple-specific deployments.

**App Intents integration exposes AI features system-wide.** Define intents using the AppIntent protocol with @Parameter annotations for inputs and perform() methods for execution. Entities represent app content objects with AppEntity conformance, while EnumerableEntityQuery or similar protocols enable Spotlight search and system-wide discovery. iOS 18 introduces assistant schemas with @AssistantIntent macros for pre-defined domains (cooking, journaling, photos), enhancing Siri's natural language understanding for domain-specific vocabulary. Implement parameterSummary properties to control how Siri phrases requests, and create AppShortcuts for discoverability in Spotlight, Shortcuts app, and Siri suggestions.

## iOS 18 delivers transformative ML capabilities for on-device AI

Core ML receives major enhancements in iOS 18 focused on **large language model support and production deployment**. The MLTensor type replaces complex MLMultiArray manipulation with high-level tensor operations similar to PyTorch, dramatically reducing boilerplate for preprocessing and postprocessing pipelines. Stateful buffers enable models to maintain persistent state across predictions, critical for LLM key-value caching that eliminates redundant computation by keeping state in GPU memory rather than transferring between CPU and GPU.

**Model compression reaches new levels of sophistication.** Block-wise quantization with 4-bit and 8-bit precision reduces model size by 8x (float32 to 4-bit) or 4x (float16 to 4-bit) while maintaining acceptable accuracy. Mistral 7B compresses from 14GB to 3.8GB using per-block linear symmetric quantization with block size 32. The compressed model runs on devices with less than 4GB available memory through stateful buffer management. Sparsity support combines with quantization for additional compression, while fused scaled dot product attention operations optimize transformer inference automatically when targeting iOS 18+.

**Neural Engine performance increased 25% on iPhone 15 Pro** without model changes, demonstrating continuous optimization at the system level. The A17 Pro Neural Engine delivers 35 trillion operations per second across 16 cores, though Apple provides no direct programming API—access occurs exclusively through Core ML framework dispatch. Core ML automatically routes operations to Neural Engine, GPU, or CPU based on operation type and model structure. Use Xcode Instruments with Points of Interest tool to verify Neural Engine utilization by filtering for "ANE" and examining H11ANE thread activity.

**Privacy manifests became mandatory May 1, 2024** for all new app submissions. The PrivacyInfo.xcprivacy property list documents four key areas: NSPrivacyTracking (tracking boolean), NSPrivacyCollectedDataTypes (data collection with purposes), NSPrivacyAccessedAPITypes (required reason APIs with approved justification codes), and NSPrivacyTrackingDomains (domains blocked when ATT permission denied). AI companion apps typically declare user IDs linked for app functionality and product personalization, file timestamp APIs for model caching (reason C617.1), and user defaults for preference storage. Xcode Organizer generates privacy reports aggregating all manifests across dependencies for App Store submissions.

**Siri integration reaches new sophistication through automatic enhancements.** Apps using standard App Intents automatically gain access to menu item invocation (Siri can trigger any app menu command), on-screen content awareness (Siri references visible text), improved conversational context (better follow-up handling), and error recovery (understands corrections). SiriKit domains for messaging, lists, media playback, and restaurant reservations receive these enhancements without code changes. For custom AI features, implement App Intents with clear parameter summaries and use assistant schemas when available for domain-specific natural language understanding.

**Live Activities and Dynamic Island provide premium real-time UI** for AI task progress and results. Define ActivityAttributes with static configuration and ContentState for dynamic updates, then implement widget extensions with Lock Screen and Dynamic Island layouts. Dynamic Island presents three states: compact (two circular areas flanking camera pill), minimal (single bubble when multiple activities active), and expanded (full-width capsule with rich content). Update activities via ActivityKit or push notifications, with streaming responses announced through proper ARIA live region equivalents. Activities timeout after 8 hours active or 12 hours on Lock Screen, making them ideal for bounded AI tasks like document analysis, image generation, or conversation summarization.

**Accessibility improvements in iOS 18 transform input modalities.** Eye Tracking uses the front-facing camera with on-device ML to provide hands-free control without additional hardware, supporting dwell control for activation and gesture simulation. Vocal Shortcuts enable custom voice commands without "Hey Siri" prefix, trained with just 3 repetitions and supporting atypical speech patterns. Listen for Atypical Speech improves recognition for users with cerebral palsy, ALS, stroke, or traumatic brain injury. VoiceOver enhancements include unlimited voice options via voice rotor, independent volume control, voice customization with EQ, configurable audio ducking, Personal Voice support, and haptic feedback options.

## Device capability detection enables intelligent model selection

**Android device classification depends on ActivityManager APIs** for memory detection and Build.VERSION.SDK_INT for Neural Networks API availability. Use ActivityManager.getMemoryInfo() to retrieve totalMem (total RAM in bytes, API 16+) and availMem (currently available RAM). The isLowRamDevice() method returns true for devices with less than 1GB RAM, reliably identifying budget devices. NNAPI availability begins at API 27 (Android 8.1) but **deprecated in Android 15**, with feature levels progressing from 1 (basic operations) through 5 (Android 12+ advanced features). Migration to TensorFlow Lite in Google Play Services or TFLite GPU delegates is essential for new implementations.

**iOS memory detection uses sysctlbyname** with "hw.memsize" to query total RAM in bytes. Neural Engine detection requires indirect methods since no public API exists for querying ANE availability. Check device model identifiers via sysctlbyname("hw.machine") against known ANE-capable devices (A12+ iPhones, A12X+ iPads, M1+ Macs). The A12 Bionic introduced **8-core Neural Engine with Core ML access** in 2018 (iPhone XS/XR), while A14 Bionic doubled to 16 cores in 2020 (iPhone 12 series). Devices with A11 or earlier (iPhone X and below) have no Neural Engine—Core ML uses CPU and GPU only.

**Implement expect/actual patterns in Kotlin Multiplatform** for clean platform separation. Define expected classes or functions in commonMain with actual implementations in androidMain and iosMain. For AI capabilities, prefer interface-based abstraction over expect classes for better testability and flexibility. Create DeviceCapabilities interface with methods like getTotalRAM(), hasNeuralAccelerator(), and getDeviceTier(), then use expect factory functions to provide platform-specific implementations. This pattern enables dependency injection and simplifies unit testing with fake implementations.

**Device tier classification drives adaptive model selection** based on quantifiable hardware metrics. Define three tiers: Flagship (8GB+ RAM, NPU present, 8+ CPU cores), Mid-range (4-8GB RAM), and Budget (<4GB RAM or marked as low RAM device). Classification logic examines total memory, neural accelerator availability, and processor count to categorize devices. The DeviceTierClassifier uses these metrics to select appropriate model variants, with safety margins of 1.5-2x the model's memory requirements to prevent out-of-memory errors.

**Model variants follow a four-tier strategy** optimized for each device capability level. Large models (512MB, FP32 precision) target flagship devices with 8GB+ RAM and hardware acceleration. Medium models (256MB, FP32) suit upper mid-range devices with 6-8GB RAM. Small models (128MB, FP16) serve mid-range devices with 4-6GB RAM and provide acceptable performance on GPU. Quantized models (64MB, INT8) ensure universal compatibility on budget devices, sacrificing some accuracy for guaranteed execution. The AdaptiveModelSelector chooses variants based on both device tier and current available memory, with fallback chains attempting progressively smaller models if loading fails.

**TensorFlow Lite GPU delegates provide hardware acceleration** on Android devices API 26+. Configure Interpreter.Options with GpuDelegate for GPU acceleration or NnApiDelegate for NNAPI (though deprecated). For devices without acceleration, set thread count (typically 4) for multi-core CPU optimization. Load models asynchronously using Dispatchers.IO to prevent blocking the UI thread, and always implement proper delegate cleanup on model disposal. Measure actual inference times on target devices across all tiers to validate performance assumptions.

**Core ML on iOS uses MLModelConfiguration** to control compute unit selection. Set computeUnits to MLComputeUnitsAll (Neural Engine + GPU + CPU) for automatic optimization, MLComputeUnitsCPUAndNeuralEngine for ANE-capable devices, or MLComputeUnitsCPUAndGPU for devices without ANE. Compile models using MLModel.compileModelAtURL() before instantiation for faster startup, caching compiled URLs. The MLProgram format (iOS 15+) provides better optimization than legacy NeuralNetwork format, enabling flexible shapes and advanced quantization techniques.

**Graceful degradation implements fallback chains** attempting optimal solutions first. The FallbackModelStrategy iterates through model variants from largest to smallest, catching OutOfMemoryError exceptions and continuing to the next variant. Before attempting to load each variant, check available memory with 50% safety margin (available ≥ model_size × 1.5) to reduce OOM likelihood. Log which variant successfully loaded for telemetry and debugging. If all variants fail, return Result.failure with descriptive error rather than crashing the app.

## WCAG 2.2 compliance requires systematic implementation and testing

**WCAG 2.2 introduced nine new success criteria** in October 2023, with particular relevance for AI applications. Focus Not Obscured (2.4.11, Level AA) requires keyboard focus indicators remain visible when focused components appear, while Target Size Minimum (2.5.8, Level AA) mandates **24×24 pixel minimum touch targets** (iOS uses 44×44 points, Android 48×48dp per platform guidelines). Accessible Authentication (3.3.8, Level AA) prevents reliance on memorization or transcription for login, crucial for AI apps using voice or biometric authentication. Redundant Entry (3.3.7, Level A) requires auto-population of previously entered information within the same session.

**Level AA conformance represents the industry standard** for accessibility compliance, requiring 4.5:1 contrast ratio for normal text (18pt+ or 14pt bold uses 3:1) and 3:1 for UI components and graphical objects. Color cannot be the only visual means of conveying information. All functionality must be keyboard accessible without specific timing for individual keystrokes. Text can resize to 200% without loss of content or functionality. These requirements apply to AI-generated content equally to static content.

**WCAG 3.0 remains in Working Draft** with expected finalization post-2026. The new structure reorganizes Guidelines → Requirements → Assertions → Methods, replacing A/AA/AAA levels with Foundational Requirements (baseline) and Supplemental Requirements (enhanced). Assertions document organizational processes and policies, while methods provide technology-specific implementation guidance. WCAG 3.0 expands scope to user agents, authoring tools, extended reality, and IoT devices. Critically, **WCAG 3.0 will coexist with WCAG 2.x**, not replace it—organizations should maintain WCAG 2.2 Level AA compliance while preparing for additional WCAG 3.0 requirements.

**iOS accessibility properties map directly to WCAG 4.1.2** (Name, Role, Value). Set accessibilityLabel for element names, accessibilityHint for action results (sparingly—most elements don't need hints), accessibilityTraits for roles and states (button, heading, selected, disabled), and accessibilityValue for current values of adjustable controls. Use isAccessibilityElement to control focus tree inclusion. VoiceOver with **70+ languages and 50+ Braille displays** works automatically with properly labeled elements. Dynamic Type support requires using system fonts and testing at all text sizes, with layouts preventing truncation at 200% zoom.

**Android accessibility services center on TalkBack** screen reader and content descriptions. Set contentDescription for non-text UI elements like icons and images. Use importantForAccessibility to control element inclusion in the accessibility hierarchy (yes/no/noHideDescendants). Mark headings with accessibilityHeading for structural navigation. Implement accessibilityLiveRegion for dynamic content announcements (polite for non-urgent, assertive for critical alerts). Form labels use labelFor to associate text labels with input fields programmatically.

**Multimodal AI presents unique accessibility challenges** requiring modal equivalence across input and output types. Text input, voice input, and keyboard navigation must be interchangeable, not forcing users into single modalities. Visual output requires audio alternatives through screen reader support and haptic feedback. Voice output needs text transcript alternatives for deaf users. Conversation pace control with pause/resume capabilities, adjustable speech rates, and interruption support prevents overwhelming users. Context preservation during mode switching (visual→audio→text transitions) maintains conversation coherence without requiring repetition.

**Screen reader optimization for AI-generated content uses ARIA live regions** and semantic structure. Mark streaming response containers with role="log" and aria-live="polite" for progressive announcement as text appears. Set aria-atomic="false" to announce only new content rather than entire messages. Use proper heading hierarchy (h1→h2→h3) for AI response structure, enabling navigation by heading in screen readers. Announce conversation turns explicitly ("You said:", "AI responded:") for context. Provide keyboard shortcuts for all voice commands, ensuring voice-only features don't create accessibility barriers.

**AI image descriptions leverage vision models** for automatic alt text generation. GPT-4 Vision, Gemini Pro Vision, and similar models provide object detection, scene understanding, and OCR capabilities for comprehensive image descriptions. Always make AI-generated descriptions **editable by content creators** for accuracy refinement. Indicate AI origin clearly: "AI-generated description:" prefix. For critical images, require human verification before publishing. This approach scales image accessibility while maintaining quality through human oversight.

**Testing methodology combines automated tools with manual verification.** axe DevTools provides **zero false positives** with 70+ WCAG test rules for web, mobile iOS, and mobile Android, integrating with Espresso, XCTest, and Appium for CI/CD pipelines. Google Accessibility Scanner offers free on-device Android testing with configurable thresholds. Xcode Accessibility Inspector provides built-in iOS auditing with VoiceOver simulation. Automated tools detect 57% of accessibility issues—**manual testing covers the remaining 43%**, including keyboard navigation, screen reader usability, and real-world assistive technology compatibility.

**Achieving 95%+ compliance requires shift-left practices** with accessibility integrated from design phase. Run automated scans daily in CI/CD pipelines, blocking pull requests with critical violations. Conduct comprehensive manual testing monthly with VoiceOver and TalkBack across multiple content types. Include users with disabilities in testing quarterly for qualitative validation. Track compliance percentage using axe DevTools reporting, prioritizing critical and serious issues (blockers like missing labels, keyboard traps) over moderate issues (focus order) and minor issues (empty headings). Generate VPAT/ACR documentation for enterprise customers demonstrating WCAG 2.2 Level AA conformance.

## Kotlin Multiplatform patterns enable feature parity across platforms

**Interface-based abstraction outperforms expect/actual classes** for platform-specific AI features. Define AIFeature interface in commonMain with methods like processInput() and isAvailable(), then implement factory functions using expect/actual pattern to provide platform-specific instances. This approach enables dependency injection, simplifies testing with mock implementations, and provides better flexibility than expect/actual classes with constructors or complex dependencies. Use Koin or Kodein for DI container management across platforms.

**Unified result models ensure consistent UX** regardless of underlying AI engine. Define common data classes for AIInferenceResult containing label, confidence, processingTimeMs, and metadata fields that both platforms populate. Create normalizer classes for each platform (MLKitNormalizer for Android, CoreMLNormalizer for iOS) that transform platform-specific result types into shared models. This abstraction layer enables switching ML frameworks without changing business logic.

**Clean architecture separates business logic from platform implementations** through domain and data layers. Use cases in commonMain define business operations (ClassifyTextUseCase) depending only on repository interfaces. Platform-specific implementations (AndroidAIRepository with ML Kit, IOSAIRepository with Core ML) reside in androidMain/iosMain and handle platform-specific framework calls. This separation enables **shared ViewModels with StateFlow** for reactive state management, with a single ViewModel serving both iOS and Android UIs while delegating to platform-optimized AI execution.

**Common test suites validate cross-platform behavior** using Kotlin's multiplatform testing support. Write tests in commonTest that execute on both platforms, verifying business logic produces expected results regardless of platform. Create fake implementations of repository interfaces for testing without real AI engines, enabling fast, deterministic unit tests. Platform-specific tests in androidTest and iosTest verify native framework integration. Implement parity tests comparing Android and iOS results for identical inputs, allowing small differences (±0.1 confidence) due to framework variations but ensuring label agreement.

**Feature flag systems handle platform-specific capabilities** gracefully. Define Feature enum listing all AI capabilities (FACE_DETECTION, ON_DEVICE_TRANSLATION, SMART_REPLY, OBJECT_TRACKING). Implement PlatformCapabilities with expect/actual providing getSupportedFeatures() listing platform-available features. The AIFeatureManager checks both platform support and remote feature flags before executing features, providing fallback mechanisms (server-side API calls) when platform capabilities lack required features. Firebase Remote Config integrates via expect/actual factory functions, enabling runtime feature control without app updates.

**Graceful degradation uses fallback chains** attempting premium features first. The AIFeatureExecutor wraps primary and fallback operations in try-catch blocks, returning Success, FallbackUsed (with reason), or Error results. For model selection, the FallbackModelStrategy iterates through variants (Large→Medium→Small→Quantized), checking memory availability before load attempts and catching OOM exceptions. Log which strategy succeeded for telemetry. Quality-based degradation adjusts inference parameters (input resolution, batch size, thread count) dynamically based on device tier.

**Directory structure follows hierarchical source sets** with clear separation. Place domain logic and models in commonMain/kotlin/domain, repository interfaces in commonMain/kotlin/data, and shared AI abstractions in commonMain/kotlin/ai. Platform implementations go in androidMain/kotlin/ai (MLKitEngine, TensorFlowLiteEngine, AndroidAIFactory) and iosMain/kotlin/ai (CoreMLEngine, VisionEngine, IOSAIFactory). For large projects with multiple AI features, modularize into separate Gradle modules (core/ai-engine, features/text-classification, features/object-detection) for better build times and separation of concerns.

**Gradle configuration manages platform-specific dependencies** cleanly. In shared/build.gradle.kts, define commonMain dependencies for coroutines and logging frameworks that work across platforms. Add androidMain dependencies for ML Kit and TensorFlow Lite artifacts from Maven Central. The iosMain source set requires no Kotlin dependencies for CoreML/Vision since these are native frameworks—access them through Kotlin/Native interop. Always specify minimum deployment targets matching your OS version decisions (iOS 15+, Android API 27+).

## Minimum OS versions balance feature access with market reach

**Android API 27 (Android 8.1 Oreo) provides optimal balance** with 96.3% market coverage as of April 2025. This version introduced NNAPI 1.0 for neural accelerator access (though deprecated in Android 15) and TensorFlow Lite with GPU delegate support via AHardwareBuffer. ML Kit works fully on API 21+, but API 27 enables hardware-accelerated inference critical for responsive AI features. **NNAPI deprecation fundamentally shifts Android ML strategy**—new implementations should use TensorFlow Lite in Google Play Services or TFLite GPU delegates rather than NNAPI.

**iOS 15 captures 96%+ market coverage** while providing Core ML 5 features. iOS 15 introduced the MLProgram model format with flexible shapes and enhanced optimization, replacing the legacy NeuralNetwork format. Devices running iOS 15+ include iPhone XS and later (A12+ with Neural Engine access), iPad Pro 2018 and later, and all Apple Silicon Macs. Going back to iOS 14 adds 2% more users but sacrifices MLProgram format benefits. Moving forward to iOS 16 loses 6% of users while adding async prediction and compute plan control—not a worthwhile trade-off for most apps.

**Android version distribution shows significant fragmentation** compared to iOS. Android 14 leads with 27.4% share, followed by Android 11 (15.9%), Android 13 (16.8%), and Android 12 (12.8%). Only 48.6% of devices run Android 13 or newer. API 27+ captures 96.3% of devices, with the excluded 3.7% running devices 7+ years old. In contrast, **iOS adoption reaches 80% within 4 months of release**, enabling more aggressive minimum version increases. iOS 18 already has 64-87% market share in Q4 2024, just months after September release.

**Feature availability matrices inform version decisions.** On Android, NNAPI Feature Level 1 (API 27) provides basic operations, Level 2 (API 28) adds expanded operations, Level 3 (API 29) introduces quality-of-service APIs and device selection, and Level 4 (API 30) enables memory domains. TensorFlow Lite GPU requires API 26+ for AHardwareBuffer support, while INT8 quantization works best on API 28+. On iOS, Core ML 2 (iOS 12) enables Neural Engine access on A12+ devices, Core ML 4 (iOS 14) adds INT8 quantization, Core ML 5 (iOS 15) introduces MLProgram format, and Core ML 7 (iOS 17) adds state management for LLMs.

**Migration strategies for existing apps** follow phased approaches. Phase 1 assesses current user base by OS version and hardware capabilities. Phase 2 implements graceful degradation with tiered models and capability detection without changing minimum version. Phase 3 (6 months later) evaluates user distribution—if <5% use old OS versions, plan minimum version increase; if >10%, continue graceful degradation. Phase 4 (12+ months) announces minimum version increases 3 months in advance with clear migration guidance before bumping requirements.

**Cost-benefit analysis reveals ROI considerations** for version selection. Choosing Android API 21 over API 27 adds 3-4% more users but increases development effort by 20-30% through additional testing, multiple model variants, and complex fallback logic. For consumer apps with alternatives, this trade-off rarely justifies the cost. For enterprise apps with captive users or critical services, broader compatibility warrants the investment. Choosing higher minimums (API 29, iOS 16) provides cleaner codebases and faster development but excludes 10-13% of potential users—acceptable when AI performance differentiates the product.

**Industry benchmarks from leading AI apps** validate these recommendations. OpenAI ChatGPT requires iOS 16.1+ and Android 8.0+ (API 26). Google Gemini targets iOS 15.0+ and Android 10+ (API 29). Adobe Photoshop with AI features needs iOS 15+ and Android 10+ (API 29). Consumer AI apps typically choose iOS 14-15 and Android 8.1-9 (API 27-28), while enterprise AI apps select iOS 15-16 and Android 10-11 (API 29-30). Bleeding-edge AI experiments use iOS 16-17 and Android 11-12 (API 30-31).

**Monitoring metrics inform future version decisions.** Track device capability distribution (percentage with hardware acceleration), performance metrics by device/OS (average inference time, 95th percentile latency), feature usage (AI feature adoption rate, drop-off by device tier), and business metrics (conversion/retention by OS version). Set concrete triggers for minimum version increases: when <10% of active users run versions below target, when <5% of new installs use old versions, or when support costs for old versions exceed revenue. Reassess minimum versions annually as market distribution shifts.

## Strategic implementation insights for production deployment

The convergence of **on-device AI capabilities with privacy-first architectures** creates unprecedented opportunities for AI companion apps, provided developers navigate platform differences strategically. Apple's Foundation Models framework democratizes access to powerful language models with zero API costs and complete offline functionality, while NNAPI deprecation on Android pushes developers toward TensorFlow Lite in Play Services for more consistent cross-platform behavior. These shifts favor Kotlin Multiplatform architectures that abstract platform specifics behind unified interfaces rather than attempting framework-level compatibility.

**Device capability detection moves from luxury to necessity** as hardware acceleration determines user experience quality. The 9x performance improvement from Neural Engine utilization on iOS and similar gains from NNAPI/GPU delegates on Android make hardware-accelerated inference the baseline expectation rather than an optimization. Apps must implement three-tier fallback strategies (flagship/mid-range/budget) with quantized model variants, accepting that users on 5+ year-old devices receive degraded experiences rather than attempting to maintain parity across all hardware. This tiered approach acknowledges that **96% market coverage at API 27/iOS 15 captures users with capable hardware**, while the excluded 4% run devices too old for acceptable AI performance regardless of optimization effort.

**Accessibility compliance transforms from checkbox exercise to architectural requirement** as AI-generated content introduces unique challenges. Screen reader optimization for streaming responses, multimodal input alternatives, and AI-assisted image descriptions require design-phase integration rather than post-development remediation. The University of Michigan's U-M GPT case study demonstrates that **accessibility-first development actually reduces time-to-market** (2 months with accessibility integrated) compared to retrofitting accessibility later. Organizations targeting enterprise markets find WCAG 2.2 Level AA compliance increasingly mandatory for procurement, making 95%+ accessibility scores business requirements rather than optional features.

The technical architecture emerging from this research prioritizes **shared business logic with platform-optimized execution** over premature abstraction. Implement Clean Architecture with use cases in commonMain depending only on repository interfaces, allowing iOS implementations to use Core ML 5 MLProgram format with Neural Engine optimization while Android implementations use TensorFlow Lite with GPU delegates. Resist the temptation to abstract away platform differences at the model format level—Core ML and TensorFlow Lite have fundamentally different optimization strategies, and forcing compatibility sacrifices performance. Instead, normalize results into common data structures and maintain feature parity through consistent UX patterns rather than identical implementation approaches.

**Testing methodology must encompass accessibility alongside functional correctness** to prevent accessibility regressions from shipping. Integrate axe DevTools into CI/CD pipelines with pull request blocking on critical violations, run monthly comprehensive screen reader testing across both VoiceOver and TalkBack, and conduct quarterly user testing with people with disabilities. The split between automated testing (57% of issues) and manual testing (43% of issues) means no tool catches everything—teams require both. Platform-specific testing on minimum-supported hardware (budget Android devices from 2018, iPhone XS) validates performance assumptions and reveals issues invisible on flagship development devices.

The path forward for Kotlin Multiplatform AI apps involves embracing platform strengths rather than fighting them. Let iOS leverage Foundation Models for text understanding while using Core ML for custom vision models. Allow Android to use TensorFlow Lite's Play Services integration for automatic optimization updates. Focus shared code on business logic, state management, and UX consistency while recognizing that optimal performance requires platform-specific tuning. This pragmatic approach delivers superior results compared to lowest-common-denominator architectures, creating AI companion apps that feel native on both platforms while maintaining a single business logic codebase.
