§ 02 Privacy & Security Architectural, not policy.
DB at rest
✓SQLCipher AES-256
256-bit random passphrase in Keystore-backed EncryptedSharedPreferences. Ship commit 99363c16. Drift-guarded by SqlCipherEncryptionTest.
Network policy
✓User-initiated only
Two allow-listed callers: HF model downloads, DuckDuckGo Instant Answer. Enforced by ManifestPrivacyTest. ADR-0006.
Package alignment
✓16KB ELF · guarded
Every .so verified ≥ 0x4000 LOAD alignment on every assemble. Commit 782d9b2c.
Analytics / telemetry
✓None · enforced at build
No Firebase, no crash reporting, no analytics SDKs. ML Kit's datatransport allow-listed with Firebase opt-out meta.
Soft delete
✓is_archived
Project + Conversation · preserves audit trail. Hard-delete is also user-initiated via the destroy path.
Retention
30 — 90 days
Pet · 30 d. Eco · 90 d. ContextSnapshot · configurable. User-owned data persists until user-deleted.

§ 04 Proposed · It's All Context Personal-document RAG + personal memory, unified.
間

Distinguishing between document, memory, fact, and observation is a design artefact — not something users care about. A pasted note, an imported PDF, a chat-derived memory, a location observation, an STT transcript, a URL body: all of them are retrievable passages of text.

One table. One retrieval path. One privacy story. The curated corpus retires. Personal-first on fresh install; RAG defaults off until the user brings their own context.

Passage
Chunk-level retrievable text from any source. One row per embedding target.
id TEXT UUID · PK
source_id TEXT FK → Source
kind TEXT pdf · md · note · memory · transcript · location
content TEXT chunk body
chunk_index INTEGER 0..N within source
embedding BLOB 384 f32 · MiniLM
token_count INTEGER budget accounting
created_at INTEGER epoch ms
deleted_at INTEGER soft delete · user-owned
pinned INTEGER retrieval priority
metadata TEXT JSON · page, lat/lng, sender, ...
Source
Owning document envelope. Lets UI list "my documents" and cascade-delete.
id TEXT UUID · PK
uri TEXT file:// · content:// · https:// · note:
kind TEXT mirrors Passage.kind
title TEXT user-facing label
byte_size INTEGER imported blob size
chunk_count INTEGER denormalised count
imported_at INTEGER epoch ms
last_accessed INTEGER for recency boost
deleted_at INTEGER soft delete · cascades
metadata TEXT JSON · author, tags, doc date

Open Questions

Q1 Unify MemoryChunk / MemoryMetadata into Passage this cycle, or keep memory separate and migrate later?
lean unify — otherwise personal memory and personal docs stay as two pipelines with parallel bugs.
Q2 Day-one source kinds — PDF on day one, or plain text + markdown + URL fetch as the minimum viable surface?
lean text + md + URL first, PDF in the second wave. Ships something real and iterable.
Q3 Embedding storage — BLOB on the Passage row, or separate PassageEmbedding table?
lean BLOB inline. Linear scan is fine at current sizes; if HNSW ever lands, extract then.
Q4 M1K3 self-identification · system prompt or pinned system-kind Passages?
lean system prompt for the ~10 identity docs. Keeps Passage = user context only.
